# 各应用市场发布接口调用链路说明

## 1. 文档目的

本文档聚焦 `app-publish-service` 中“发布到各应用市场”的实际调用链路，重点说明：

- 从业务入口到商店发布器的公共编排逻辑
- 各商店发布接口的调用顺序、关键前置查询和审核回查逻辑
- 各商店 `storeReleaseId`、token、灰度发布能力、审核状态判定方式

本文档对应的是当前代码实现，不是第三方官方接口字段手册的替代。

---

## 2. 相关入口

### 2.1 提审入口

- `POST /api/releases/submit`
  - 入口控制器：`ReleaseController.submit`
  - 核心服务：`ReleaseOrchestrationService.submit`

### 2.2 CI/CD 一键触发入口

- `POST /api/cicd/releases/trigger`
  - 入口控制器：`CicdController.trigger`
  - 核心服务：`CicdReleaseService.trigger`
  - 说明：先走包准备，再复用 `ReleaseOrchestrationService.submit`

### 2.3 审核回查入口

- 定时任务：`ReleaseReviewPollingJob.poll`
  - 调用 `ReleaseOrchestrationService.pollAuditResults`
  - 只轮询状态为 `AUDITING` 的发布记录

---

## 3. 公共发布主链路

## 3.1 submit 主链路

`/api/releases/submit` 的统一执行顺序如下：

1. `ReleaseOrchestrationService.submit`
2. `PackageVersionService.requireVersion`
3. 校验 `releaseType`
   - `1` 表示全量发布
   - `2` 表示灰度发布
4. 遍历请求中的 `storeTypes`
5. `AppManagementService.requireStoreConfig`
6. 创建 `app_release_record`
   - 初始状态：`DOWNLOADING`
7. `PackageVersionService.prepareVersionForSubmit`
   - 统一准备提交用安装包
   - 失败时状态更新为 `DOWNLOAD_FAIL`
8. 每个商店依次执行
   - 状态切换为 `API_PENDING`
   - `TokenService.getValidToken`
   - `ConfigurableStorePublisher.submitRelease`
   - 成功则写入 `storeReleaseId`、请求/响应日志，状态改为 `AUDITING`
   - 失败则状态改为 `REJECT`

对应的发布器选择逻辑：

- `ConfigurableStorePublisher`
  - `VivoStorePlatformPublisher`
  - `OppoStorePlatformPublisher`
  - `HuaweiStorePlatformPublisher`
  - `XiaomiStorePlatformPublisher`
  - `SanxingStorePlatformPublisher`
  - `YingyongbaoStorePlatformPublisher`

## 3.2 queryReview 主链路

定时轮询审核状态时，统一执行顺序如下：

1. `ReleaseOrchestrationService.pollAuditResults`
2. 查询所有 `AUDITING` 记录
3. 对每条记录：
   - `AppManagementService.requireStoreConfig`
   - `TokenService.getValidToken`
   - `ConfigurableStorePublisher.queryReview`
4. 将商店返回结果映射为内部状态
   - `PASS`
   - `AUDITING`
   - `REJECT`
   - `OFFLINE`
5. 如果已结束，写入 `finishTime`

## 3.3 token 获取公共逻辑

统一由 `TokenService.getValidToken` 负责：

1. 先查 `app_api_token_cache`
2. 未命中或临近过期时调用 `StorePublisher.refreshToken`
3. 写回缓存后返回 token

当前实现里，token 分两类：

- `ACCESS_TOKEN`
  - 由商店 OAuth/JWT 接口换取
- `STATIC`
  - 实际并不依赖远端 token 接口，更多是用配置字段当占位凭证

## 3.4 请求日志落库

各商店接口的真实调用都会尽量通过 `executeStoreRequest(...)` 执行，并配合：

- `StoreRequestTrace`
- `StoreRequestLogContextHolder`
- `StoreRequestLogService`

将请求动作、HTTP 方法、URL、请求参数、响应内容、耗时写入日志。  
此外，提审成功后还会把聚合后的 `apiRequestLog`、`apiResponseLog` 写回 `app_release_record`。

## 4. 灰度发布支持矩阵

系统在 `ReleaseOrchestrationService.validateReleaseRequest` 中对灰度发布做了显式限制。

| 商店 | 是否支持 `releaseType=2` | 说明 |
| --- | --- | --- |
| vivo | 是 | 走分阶段发布链路 |
| huawei | 是 | 提交和查询都带 `releaseType=3` 的分发语义 |
| sanxing | 是 | 额外调用 `stagedRolloutRate` |
| oppo | 否 | 代码中直接禁止 |
| xiaomi | 否 | 代码中直接禁止 |
| yingyongbao | 否 | 代码中直接禁止 |

## 5. 各商店调用链路

## 5.1 vivo

### token 逻辑

- `refreshToken` 不请求远端接口
- 直接返回 `STATIC` token
- token 值取 `clientId`，取不到则退化为固定标记

### submit 链路

入口：

- `VivoStorePlatformPublisher.submitRelease`
- `submitVivoRelease`

调用顺序：

1. `ensureVivoSubmitReady`
   - 先调用 `app.query.details`
   - 判断应用是否已存在
2. 分支处理
   - 不存在：`submitVivoCreateApp`
   - 已存在且 `releaseType=2`：`submitVivoStageRelease`
   - 已存在且 `releaseType=1`：`submitVivoFullRelease`
3. 包上传
   - `uploadVivoPackageBundle`
   - 32 位包：`app.upload.apk.app.32`
   - 64 位包：`app.upload.apk.app.64`
4. 如果是新建应用，还会补充图标、截图等素材上传
5. 提交接口
   - 全量更新：`app.sync.update.subpackage.app`
   - 灰度发布：`app.sync.create.update.stage.app`
   - 新建应用：`app.sync.create.subpackage.app`

实现特点：

- vivo 是“先查详情，再决定创建/更新/灰度”的模式
- 上传和提交分多个接口完成
- `storeReleaseId` 最终用上传结果中的序列号

### queryReview 链路

入口：

- `queryReview`

调用顺序：

1. 如果 `releaseType=2`
   - 调 `app.query.stage.details`
   - 用 `auditStatus + effectStatus` 映射内部状态
2. 否则
   - 调 `app.query.details`
   - 用 `status + saleStatus + rejectReason` 映射内部状态

### 审核状态判定

- 灰度发布：`mapVivoStageStatus`
- 全量发布：`mapVivoAppStatus`

---

## 5.2 oppo

### token 逻辑

- 通过 `GET /developer/v1/token` 刷新
- 依赖 `clientId + clientSecret`
- 返回 `ACCESS_TOKEN`

### submit 链路

入口：

- `OppoStorePlatformPublisher.submitRelease`
- `submitOppoRelease`

调用顺序：

1. 查询应用多版本信息
   - `POST /resource/v1/app/multi-info`
2. 从返回结果中抽取当前应用已有元信息
   - `resolveOppoVersionInfo`
   - `buildOppoSubmitPayload`
3. 逐个安装包执行上传前握手
   - `POST /resource/v1/upload/get-upload-url`
4. 上传包文件
   - `uploadOppoPackage`
5. 汇总为 `apk_url` 列表后提交
   - `POST /resource/v1/app/upd`

实现特点：

- oppo 不直接从本地元数据拼完整提审参数
- 而是先查商店已有应用资料，再补齐提交字段
- 多架构包会分别上传，再统一提交
- `storeReleaseId` 优先取 `task_id`

### queryReview 链路

入口：

- `queryOppoReview`

调用顺序：

1. 组装查询参数
   - `pkg_name`
   - `version_code`
2. 调 `POST /resource/v1/app/task-state`
3. `mapOppoStatus` 映射内部状态

### 审核状态判定

- 主要依据 `task_state`
- 驳回时尝试读取 `err_msg/msg/message`

---

## 5.3 huawei

### token 逻辑

- 通过 `POST /oauth2/v1/token` 获取
- 使用 `grant_type=client_credentials`
- 依赖 `clientId + clientSecret`

### submit 链路

入口：

- `HuaweiStorePlatformPublisher.submitRelease`
- `submitHuaweiRelease`

调用顺序：

1. 查询 `appId`
   - `GET /publish/v2/appid-list`
2. 解析待提交包列表
   - 支持 32 位、64 位或单包
3. 对每个包执行：
   - 获取上传地址：`GET /publish/v2/upload-url/for-obs`
   - 上传文件到 OBS：`uploadHuaweiFile`
4. 更新应用文件信息
   - `PUT /publish/v2/app-file-info`
5. 提交审核
   - `POST /publish/v2/app-submit`

实现特点：

- huawei 采用“先拿 OBS 上传地址，再直传文件，再更新 file info，再 submit”的四段式流程
- `storeReleaseId` 直接使用 `appId`
- 灰度与全量提交都复用同一主链路，但 `releaseType` 会影响请求参数

### queryReview 链路

入口：

- `queryHuaweiReview`

调用顺序：

1. 优先从 `record.storeReleaseId` 取 `appId`
2. 如果没有，则再次调用 `appid-list` 反查
3. 调 `GET /publish/v2/app-info`
   - 全量：`releaseType=1`
   - 灰度：`releaseType=3`
4. 读取
   - `appInfo`
   - `auditInfo`
   - `phasedReleaseInfo`
5. 状态映射
   - 全量：`mapHuaweiReleaseStatus`
   - 灰度：`mapHuaweiStagedStatus`

### 审核状态判定

- 驳回原因优先取 `auditOpinion` 及版权/备案相关审核意见
- 灰度场景会额外参考 `phasedReleaseInfo`

---

## 5.4 xiaomi

### token 逻辑

- 不请求远端 token 接口
- 直接返回 `STATIC`
- token 值用 `email`，取不到则用固定标记

### submit 链路

入口：

- `XiaomiStorePlatformPublisher.submitRelease`
- `submitXiaomiRelease`

调用顺序：

1. 解析 `userName`
   - 优先 `email`
   - 其次 `accountName`
2. 本地解析包路径、第二 APK、项目发布元数据
3. 先调用包查询接口
   - `POST /dev/query`
4. `resolveXiaomiSubmitContext`
   - 根据 `create/updateVersion` 判断是创建还是更新
   - 组装 `appInfo`
   - 校验 icon、截图、pad 截图、privacyUrl 等
5. 组装 `RequestData`
6. 对 `RequestData + 文件` 生成 `SIG`
7. 调提交接口
   - `POST /dev/push`
   - `multipart/form-data`

实现特点：

- 小米是“先 `/dev/query`，再 `/dev/push`”
- 是否新建、是否允许更新版本，完全依赖 `/dev/query` 返回值
- 提交请求是多段表单，包含 APK、secondApk、icon、截图等文件
- `storeReleaseId` 使用 `packageName:versionCode`

### queryReview 链路

入口：

- `queryXiaomiReview`

调用顺序：

1. 再次调用 `POST /dev/query`
2. 读取 `packageInfo`
3. 状态判定分两层：
   - 先看是否存在明确终态
     - `reject/rejected/failed` -> `REJECT`
     - `offline` -> `OFFLINE`
   - 否则按 `packageInfo.versionCode` 与目标版本 `V_NEW` 比较

### 审核状态判定

当前实现已经按版本号比较重构为：

- `packageInfo.versionCode >= V_NEW`
  - 判定为 `PASS`
  - 含义：系统认为新版已经审核完成，商店已切到新版对外分发
- `packageInfo.versionCode < V_NEW`
  - 判定为 `AUDITING`
  - 含义：新包已提审，但商店当前仍在跑旧版

补充说明：

- 比较结果会写入 `apiResponseLog`
- 包括 `targetVersionCode`、`packageVersionCode`、比较值等信息，便于排查

## 5.5 sanxing

### token 逻辑

- 通过 JWT 换取 `ACCESS_TOKEN`
- 依赖服务账号信息和私钥
- 支持两种换 token 模式
  - body 传 `assertion`
  - 或 `Authorization: Bearer <jwt>`

### submit 链路

入口：

- `SanxingStorePlatformPublisher.submitRelease`
- `submitSanxingRelease`

调用顺序：

1. `resolveSanxingContext`
   - 得到 `contentId`、包路径、是否 GMS 等
2. 查询当前内容信息
   - `GET /seller/contentInfo`
3. 创建上传会话
   - `POST /seller/createUploadSessionId`
4. 上传二进制文件
   - `POST https://seller.samsungapps.com/galaxyapi/fileUpload`
5. 追加二进制版本
   - `POST /seller/v2/content/binary`
6. 更新内容元数据
   - `POST /seller/contentUpdate`
7. 如果是灰度发布
   - `PUT /seller/v2/content/stagedRolloutRate`
8. 最后提交审核
   - `POST /seller/contentSubmit`

实现特点：

- sanxing 先查 `contentInfo`，再创建 upload session，再文件上传，再挂载 binary，再更新内容，再 submit
- 灰度发布是单独的附加步骤，不和普通 submit 混在一起
- `storeReleaseId` 使用 `contentId`

### queryReview 链路

入口：

- `querySanxingReview`

调用顺序：

1. 根据发布记录解析 `contentId`
2. 调 `GET /seller/contentInfo`
3. 根据 `contentStatus/appStatus/status` 映射内部状态
4. 若状态不明确，则检查 `binaryList.versionCode` 是否已命中目标版本

### 审核状态判定

- 主映射：`mapSanxingStatus`
- 兜底判定：`sanxingBinaryMatchesTargetVersion`
  - 如果商店当前 binary 版本已等于目标版本，判定为 `PASS`
  - 否则判定为 `AUDITING`

## 5.6 yingyongbao

### token 逻辑

- 不请求远端 token 接口
- 直接返回 `STATIC`
- token 值优先取 `clientId`

### submit 链路

入口：

- `YingyongbaoStorePlatformPublisher.submitRelease`
- `submitYingyongbaoRelease`

调用顺序：

1. `resolveYingyongbaoContext`
   - 解析 `packageName`、`appId`、项目元数据
2. 查询应用详情
   - `POST /query_app_detail`
3. 构造更新 payload
   - `buildYingyongbaoUpdatePayload`
   - 处理 `deploy_type`
   - 处理定时发布时间 `deploy_time`
4. APK 上传前先获取上传信息
   - `POST /get_file_upload_info`
5. 按返回的 `pre_sign_url` 执行文件上传
   - `PUT pre_sign_url`
6. 汇总文件序列号和 md5 后提交更新
   - `POST /update_app`

实现特点：

- 应用宝采用“先拿上传信息，再直传文件，再回调 update_app”的模式
- 32/64 位包分别上传
- `storeReleaseId` 使用 `appId`

### queryReview 链路

入口：

- `queryYingyongbaoReview`

调用顺序：

1. 解析 `packageName` 和 `appId`
2. 调 `POST /query_app_update_status`
3. 使用 `audit_status` 映射内部状态

### 审核状态判定

- 通过 `mapYingyongbaoAuditStatus`
- 驳回原因优先取 `audit_reason`

## 6. 商店差异对比

| 商店 | token 类型 | submit 前置查询 | 文件上传方式 | 提审接口 | 审核查询接口 | `storeReleaseId` |
| --- | --- | --- | --- | --- | --- | --- |
| vivo | `STATIC` | `app.query.details` | 先上传 APK，再提交 | `app.sync.*` | `app.query.details` / `app.query.stage.details` | 上传序列号 |
| oppo | `ACCESS_TOKEN` | `app/multi-info` | 先申请上传 URL，再上传 | `app/upd` | `app/task-state` | `task_id` |
| huawei | `ACCESS_TOKEN` | `appid-list` | 先取 OBS URL，再直传 | `app-submit` | `app-info` | `appId` |
| xiaomi | `STATIC` | `/dev/query` | multipart 直接带文件 | `/dev/push` | `/dev/query` | `packageName:versionCode` |
| sanxing | `ACCESS_TOKEN` | `contentInfo` | 先建 session，再上传 | `contentSubmit` | `contentInfo` | `contentId` |
| yingyongbao | `STATIC` | `query_app_detail` | 先取 `pre_sign_url`，再 PUT 上传 | `update_app` | `query_app_update_status` | `appId` |

## 7. 状态流转总结

发布记录的典型状态流转如下：

1. `DOWNLOADING`
2. `DOWNLOAD_SUCCESS`
3. `API_PENDING`
4. `AUDITING`
5. 终态之一
   - `PASS`
   - `REJECT`
   - `OFFLINE`

异常分支：

- 包准备失败：`DOWNLOAD_FAIL`
- 商店提交异常：直接转 `REJECT`

---

## 8. 结论

当前系统的设计特点是：

1. 公共编排统一
   - 统一由 `ReleaseOrchestrationService + TokenService + ConfigurableStorePublisher` 驱动
2. 商店差异下沉
   - 每个商店单独实现 `submitRelease/queryReview/refreshToken`
3. 审核回查统一入口
   - 统一由定时任务轮询，再回落到商店级状态映射
4. 灰度能力不是全商店通用能力
   - 当前仅 vivo、huawei、sanxing 走 API 灰度链路
5. 请求和聚合日志都保留
   - 方便排查第三方接口失败、字段缺失、状态映射异常等问题
