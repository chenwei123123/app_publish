# 各应用市场发布接口调用链路说明

## 1. 文档目的

本文档对应当前 `app-publish-service` 代码实现，说明“发布到各应用市场”的实际调用链路，重点覆盖：

- 统一发布编排入口与状态流转
- token 获取与缓存逻辑
- 各市场 `submitRelease` / `queryReview` / `refreshToken` 的调用顺序
- 各市场的 `storeReleaseId` 取值、灰度支持能力、审核状态映射方式

说明：

- 本文档描述的是当前项目代码行为，不是第三方官方接口文档的逐字段翻译
- 如与三方官方文档存在差异，以当前项目实现为准

---

## 2. 相关入口

### 2.1 提审入口

- `POST /api/releases/submit`
  - 控制器：`ReleaseController.submit`
  - 核心服务：`ReleaseOrchestrationService.submit`

### 2.2 CI/CD 一键触发入口

- `POST /api/cicd/releases/trigger`
  - 控制器：`CicdController.trigger`
  - 核心服务：`CicdReleaseService.trigger`
  - 说明：先调用 `PackageVersionService.upload` 上传安装包并生成版本，再复用 `ReleaseOrchestrationService.submit`

### 2.3 审核轮询入口

- 定时任务：`ReleaseReviewPollingJob.poll`
  - 调用 `ReleaseOrchestrationService.pollAuditResults`
  - 仅轮询状态为 `AUDITING` 的发布记录
  - 调度周期：`${app.review-poll-delay-ms:60000}`

---

## 3. 公共发布主链路

## 3.1 submit 主链路

`/api/releases/submit` 的统一执行顺序如下：

1. `ReleaseOrchestrationService.submit`
2. 进入事务准备阶段 `prepareSubmit`
3. `PackageVersionService.requireVersion`
4. 解析 `releaseMode`
   - 支持 `manual`
   - 支持 `api`
5. 规范化并校验 `releaseType`
   - 默认值为 `1`
   - `1` 表示全量发布
   - `2` 表示灰度发布
6. 遍历请求中的 `storeTypes`
   - `AppManagementService.requireStoreConfig`
   - 校验 `storeConfig.apiStatus == 1`
7. 为每个商店创建一条 `app_release_record`
   - 初始状态：`DOWNLOADING`
   - 记录 `releaseMode`、`releaseType`、灰度参数、版本信息、应用信息
8. 统一准备提审包
   - `PackageVersionService.prepareVersionForSubmit`
   - 成功后所有记录状态更新为 `DOWNLOAD_SUCCESS`
   - 失败则所有记录状态更新为 `DOWNLOAD_FAIL`
9. 对每个商店并发提交
   - `scheduleStoreSubmit`
   - 实际通过 `releaseSubmitExecutor` + `CompletableFuture` 并发执行
10. 单个商店提交流程 `submitToStore`
    - 状态切换为 `API_PENDING`
    - `TokenService.getValidToken`
    - `ConfigurableStorePublisher.submitRelease`
    - 成功时写入：
      - `storeReleaseId`
      - `apiRequestLog`
      - `apiResponseLog`
      - `releaseTime`
      - 状态改为 `AUDITING`
    - 失败时写入：
      - `rejectReason`
      - `apiResponseLog`
      - `finishTime`
      - 状态改为 `REJECT`

补充说明：

- 旧文档中的“每个商店依次执行”已不再准确，当前实现是“统一备包后并发提交各商店”
- 商店发布能力由 `ConfigurableStorePublisher` 分发到具体 `StorePlatformPublisher`

当前注册的市场发布器如下：

- `VivoStorePlatformPublisher`
- `OppoStorePlatformPublisher`
- `HuaweiStorePlatformPublisher`
- `RongyaoStorePlatformPublisher`
- `XiaomiStorePlatformPublisher`
- `SanxingStorePlatformPublisher`
- `YingyongbaoStorePlatformPublisher`

## 3.2 queryReview 主链路

定时轮询审核状态时，统一执行顺序如下：

1. `ReleaseReviewPollingJob.poll`
2. `ReleaseOrchestrationService.pollAuditResults`
3. 查询所有 `AUDITING` 状态记录
4. 对每条记录执行：
   - `AppManagementService.requireStoreConfig`
   - `TokenService.getValidToken`
   - `ConfigurableStorePublisher.queryReview`
5. 将商店返回结果映射为内部状态：
   - `PASS`
   - `AUDITING`
   - `REJECT`
   - `OFFLINE`
6. 若状态发生变化则更新：
   - `releaseStatus`
   - `rejectReason`
   - `apiResponseLog`
   - 如状态结束则写入 `finishTime`
7. 即使状态未变化，也会更新最新 `apiResponseLog`

## 3.3 token 获取公共逻辑

统一由 `TokenService.getValidToken` 负责：

1. 根据商店类型推导预期 token 类型
2. 以 `storeConfigId:tokenType` 为粒度加本地锁
3. 查询 `app_api_token_cache` 中最近一条缓存
4. 若缓存未过期，直接返回
   - 过期判断阈值：`now + app.tokenRefreshAheadSeconds`
5. 若缓存不存在或即将过期，则调用 `storePublisher.refreshToken`
6. 将最新 token 写回缓存表并返回

当前项目使用的 token 类型：

- `ACCESS_TOKEN`
  - 通过远端 OAuth/JWT/签名接口换取
- `STATIC`
  - 不依赖远端 token 接口，更多是配置字段占位

`TokenService.expectedTokenType` 的当前判断规则：

- 以下商店固定按 `STATIC` 处理
  - `vivo`
  - `xiaomi`
  - `yingyongbao`
- 只要 `storeConfig.token` 有值，也按 `STATIC`
- 其余默认按 `ACCESS_TOKEN`

## 3.4 请求日志落库

商店接口调用统一尽量通过 `executeStoreRequest(...)` 执行，并结合：

- `StoreRequestTrace`
- `StoreRequestLogContextHolder`
- `StoreRequestLogService`

会记录：

- action
- HTTP 方法
- URL
- query / form / json 请求参数
- 响应内容
- 成功失败状态
- 耗时

此外，提审成功后还会将聚合后的：

- `apiRequestLog`
- `apiResponseLog`

回写到 `app_release_record`，用于后续排障。

## 4. 灰度发布支持矩阵

系统在 `ReleaseOrchestrationService.validateReleaseRequest` 中显式限制了部分商店的灰度提审能力。

当 `releaseType=2` 时还会统一校验：

- `grayPercent` 必填
- `grayPercent` 范围必须在 `1-99`
- `grayStartTime` / `grayEndTime` 必填
- `grayStartTime < grayEndTime`

当前支持情况如下：

| 商店 | 是否支持 `releaseType=2` | 说明 |
| --- | --- | --- |
| vivo | 是 | 走 `app.sync.create.update.stage.app` |
| huawei | 是 | 提交和查询都带 `releaseType=3` 语义 |
| rongyao | 是 | submit body 中构造 `phasedReleaseInfo` |
| sanxing | 是 | 额外调用 `PUT /seller/v2/content/stagedRolloutRate` |
| oppo | 否 | 代码中直接禁止 |
| xiaomi | 否 | 代码中直接禁止 |
| yingyongbao | 否 | 代码中直接禁止 |

## 5. 各商店调用链路

## 5.1 vivo

### token 逻辑

- 不请求远端 token 接口
- 直接返回 `STATIC`
- token 值优先取 `clientId`

### submit 链路

入口：

- `VivoStorePlatformPublisher.submitRelease`
- `submitVivoRelease`

调用顺序：

1. `ensureVivoSubmitReady`
   - 先查 `app.query.details`
   - 判断应用是否已存在
2. 分支处理
   - 不存在：`submitVivoCreateApp`
   - 已存在且 `releaseType=2`：`submitVivoStageRelease`
   - 已存在且 `releaseType=1`：`submitVivoFullRelease`
3. 包上传
   - `uploadVivoPackageBundle`
   - 32 位：`app.upload.apk.app.32`
   - 64 位：`app.upload.apk.app.64`
4. 新建应用场景还会补充素材上传
   - 图标：`app.upload.icon`
   - 截图：`app.upload.screenshot`
5. 调用提交接口
   - 新建：`app.sync.create.subpackage.app`
   - 全量更新：`app.sync.update.subpackage.app`
   - 灰度发布：`app.sync.create.update.stage.app`

实现特点：

- vivo 是“先查详情，再决定新建 / 更新 / 灰度”的模式
- 32/64 位包分开上传
- 灰度提交通常会带 `stagedStartTime`、`stagedEndTime`、`stagedProportion`
- `storeReleaseId` 取上传成功返回的 `serialnumber`

### queryReview 链路

入口：

- `queryReview`

调用顺序：

1. 如果 `releaseType=2`
   - 查 `app.query.stage.details`
   - 根据 `auditStatus + effectStatus` 映射状态
2. 否则
   - 查 `app.query.details`
   - 根据 `status + saleStatus + rejectReason` 映射状态

状态判定：

- 灰度：`mapVivoStageStatus`
- 全量：`mapVivoAppStatus`

---

## 5.2 oppo

### token 逻辑

- `GET /developer/v1/token`
- 依赖 `clientId + clientSecret`
- 返回 `ACCESS_TOKEN`

### submit 链路

入口：

- `OppoStorePlatformPublisher.submitRelease`
- `submitOppoRelease`

调用顺序：

1. 查询应用详情上下文
   - 先调 `GET /resource/v1/app/multi-info`
   - 若返回 `data` 为空，则回退调 `GET /resource/v1/app/info`
   - 代码入口：`queryOppoAppDetail`
2. 组装提交参数
   - 多包场景从 `apk_info` 中取版本信息
   - 普通包场景直接使用 `/resource/v1/app/info` 返回的平铺 `data`
   - `resolveOppoVersionInfo`
   - `buildOppoSubmitPayload`
3. 逐个包申请上传配置
   - `GET /resource/v1/upload/get-upload-url`
4. 上传安装包
   - `uploadOppoPackage`
5. 汇总成 `apk_url` 列表并提交
   - `POST /resource/v1/app/upd`

实现特点：

- oppo 不直接从本地元数据拼完整提审参数
- 而是优先查询商店侧已有资料后再补齐提交字段
- 多架构包分别上传，再统一提交
- 请求日志固定记录 `multiInfo`，回退时额外记录 `info`
- `storeReleaseId` 优先取 `task_id`，取不到则退化为 `packageName:versionCode`

### queryReview 链路

入口：

- `queryOppoReview`

调用顺序：

1. 组装查询参数
   - `pkg_name`
   - `version_code`
2. `POST /resource/v1/app/task-state`
3. `mapOppoStatus` 映射内部状态

状态判定：

- 主要依据 `task_state`
  - `-1` -> `REJECT`
  - `1` -> `AUDITING`
  - `2` -> `PASS`
  - `3` -> `OFFLINE`
- 驳回原因优先读取 `err_msg/msg/message`

---

## 5.3 huawei

### token 逻辑

- `POST /oauth2/v1/token`
- `grant_type=client_credentials`
- 依赖 `clientId + clientSecret`
- 返回 `ACCESS_TOKEN`

### submit 链路

入口：

- `HuaweiStorePlatformPublisher.submitRelease`
- `submitHuaweiRelease`

调用顺序：

1. 通过包名查询 `appId`
   - `GET /publish/v2/appid-list`
2. 解析待上传安装包
   - 当前实现实际要求 `package_url_64`
3. 为每个包获取上传地址
   - `GET /publish/v2/upload-url/for-obs`
4. 直传文件到 OBS
   - `uploadHuaweiFile`
5. 更新应用文件信息
   - `PUT /publish/v2/app-file-info`
6. 提交审核
   - `POST /publish/v2/app-submit`

实现特点：

- huawei 是典型“先拿上传地址，再直传，再更新 file info，再 submit”
- `storeReleaseId` 直接使用 `appId`
- 灰度场景会在提交参数里带：
  - `releaseType=3`
  - `phasedReleaseStartTime`
  - `phasedReleaseEndTime`
  - `phasedReleasePercent`
  - `phasedReleaseDescription`

### queryReview 链路

入口：

- `queryHuaweiReview`

调用顺序：

1. 通过包名反查 `appId`
2. `GET /publish/v2/app-info`
   - 全量：`releaseType=1`
   - 灰度：`releaseType=3`
3. 读取返回结构
   - `appInfo`
   - `auditInfo`
   - `phasedReleaseInfo`
4. 状态映射
   - 全量：`mapHuaweiReleaseStatus`
   - 灰度：`mapHuaweiStagedStatus`

状态判定：

- 全量主要看 `appInfo.releaseState`
- 灰度先看全量基础状态，再看 `phasedReleaseInfo.state`
  - `RELEASE` -> `PASS`
  - `SUSPEND` / `CANCEL` -> `OFFLINE`
- 驳回原因优先取：
  - `auditOpinion`
  - `copyRightAuditOpinion`
  - `copyRightCodeAuditOpinion`
  - `recordAuditOpinion`

## 5.4 rongyao

### token 逻辑

- `POST https://iam.developer.honor.com/auth/token`
- 表单模式 `grant_type=client_credentials`
- 依赖 `clientId + clientSecret`
- 返回 `ACCESS_TOKEN`

### submit 链路

入口：

- `RongyaoStorePlatformPublisher.submitRelease`
- `submitRongyaoRelease`

调用顺序：

1. 根据包名查询 `appId`
   - `GET /openapi/v1/publish/get-app-id`
2. 解析安装包
   - 当前实现实际要求 `package_url_64`
3. 申请文件上传地址
   - `POST /openapi/v1/publish/get-file-upload-url`
4. 上传文件
   - `POST /openapi/v1/publish/file-upload`
5. 更新文件信息
   - `POST /openapi/v1/publish/update-file-info`
6. 提交审核
   - `POST /openapi/v1/publish/submit-audit`

实现特点：

- 荣耀整体结构与华为接近，但接口路径不同
- `storeReleaseId` 取 `submit-audit` 返回的 `data`，即发布单 `releaseId`
- 灰度场景提交体中会带：
  - `releaseType=3`
  - `phasedReleaseInfo.releasePercentage`
  - `phasedReleaseInfo.releaseStartDate`
  - `phasedReleaseInfo.releaseEndDate`

### queryReview 链路

入口：

- `queryRongyaoReview`

调用顺序：

1. 校验 `packageName` 与 `storeReleaseId`
2. 根据包名查询 `appId`
3. 查询审核结果
   - `POST /openapi/v1/publish/get-audit-result`
4. 如果是灰度发布，优先查询灰度状态
   - `GET /openapi/v1/publish/get-phased-release-info`
5. 必要时回查当前发布信息
   - `GET /openapi/v1/publish/get-app-current-release`

状态判定：

- `auditResult`
  - `1` -> `PASS`
  - `2` -> `REJECT`
  - `0` / `4` -> `AUDITING`
  - `3` -> 灰度时按 `PASS`，非灰度按 `AUDITING`
- 灰度补充判定 `releaseStatus`
  - `1` / `2` / `4` -> `PASS`
  - `3` / `5` -> `OFFLINE`
- 驳回原因优先取 `auditMessage`

## 5.5 xiaomi

### token 逻辑

- 不请求远端 token 接口
- 直接返回 `STATIC`
- token 值优先取 `email`

### submit 链路

入口：

- `XiaomiStorePlatformPublisher.submitRelease`
- `submitXiaomiRelease`

调用顺序：

1. 解析 `userName`
   - 优先 `email`
   - 其次 `accountName`
2. 解析本地包、第二 APK、项目元数据
3. 先查包状态
   - `POST /dev/query`
4. 构建提交上下文
   - `resolveXiaomiSubmitContext`
   - 根据 `create / updateVersion` 决定新建或更新
   - 组装 `appInfo`
   - 校验 icon、截图、pad 截图、privacyUrl 等
5. 组装 `RequestData`
6. 对 `RequestData + 文件列表` 生成 `SIG`
7. 提交
   - `POST /dev/push`
   - `multipart/form-data`

实现特点：

- 小米是“先 `/dev/query`，再 `/dev/push`”
- 提交时会携带：
  - `apk`
  - 可选 `secondApk`
  - `icon`
  - 多张 `screenshot_*`
  - 可选 `screenshot_pad_*`
- `storeReleaseId` 固定使用 `packageName:versionCode`

### queryReview 链路

入口：

- `queryXiaomiReview`

调用顺序：

1. 再次调用 `POST /dev/query`
2. 读取 `packageInfo`
3. 优先尝试识别显式终态
   - `status/reviewStatus/auditStatus/appStatus/state`
4. 若没有显式终态，再按版本号比较
   - 比较商店返回版本号与目标版本号

状态判定：

- 显式终态：
  - `reject` / `rejected` / `failed` -> `REJECT`
  - `offline` -> `OFFLINE`
- 否则比较 `packageInfo.versionCode` 与目标版本
  - `>= target` -> `PASS`
  - `< target` -> `AUDITING`
- 驳回原因优先读：
  - `rejectReason`
  - `reason`
  - `auditReason`
  - `message`

## 5.6 sanxing

### token 逻辑

- 先生成 JWT
- 再请求 `POST /auth/accessToken`
- 支持两种换 token 模式
  - body 里传 `assertion`
  - 或 `Authorization: Bearer <jwt>`
- 返回 `ACCESS_TOKEN`

### submit 链路

入口：

- `SanxingStorePlatformPublisher.submitRelease`
- `submitSanxingRelease`

调用顺序：

1. 解析提交上下文
   - 获取 `contentId`
   - 获取安装包路径
2. 查询当前内容详情
   - `GET /seller/contentInfo`
3. 创建上传会话
   - `POST /seller/createUploadSessionId`
4. 上传二进制文件
   - `POST https://seller.samsungapps.com/galaxyapi/fileUpload`
5. 追加 binary
   - `POST /seller/v2/content/binary`
6. 更新内容元数据
   - `POST /seller/contentUpdate`
7. 灰度发布时额外设置比例
   - `PUT /seller/v2/content/stagedRolloutRate`
8. 最终提交审核
   - `POST /seller/contentSubmit`

实现特点：

- 三星是“先查 contentInfo，再建上传 session，再传文件，再挂 binary，再更新内容，再 submit”
- `storeReleaseId` 使用 `contentId`
- 灰度提交依赖额外的 `stagedRolloutRate` 调用

### queryReview 链路

入口：

- `querySanxingReview`

调用顺序：

1. 解析 `contentId`
   - 优先 `record.storeReleaseId`
   - 其次 `storeConfig.appId`
   - 再其次发布元数据中的 `sanxing.contentId`
2. 查询 `GET /seller/contentInfo`
3. 根据 `contentStatus/appStatus/status` 做主映射
4. 如果状态不明确，兜底比较 `binaryList.versionCode` 是否命中目标版本

状态判定：

- 主映射函数：`mapSanxingStatus`
- 兜底函数：`sanxingBinaryMatchesTargetVersion`
  - 命中目标版本通常判为 `PASS`
  - 否则保持 `AUDITING`
- 驳回原因优先读：
  - `reviewComment`
  - `reviewMessage`
  - `message`

## 5.7 yingyongbao

### token 逻辑

- 不请求远端 token 接口
- 直接返回 `STATIC`
- token 值优先取 `clientId`

### submit 链路

入口：

- `YingyongbaoStorePlatformPublisher.submitRelease`
- `submitYingyongbaoRelease`

调用顺序：

1. 解析上下文
   - 获取 `appId`
   - 获取 `packageName`
   - 获取元数据
2. 查询应用详情
   - `POST /query_app_detail`
3. 构造更新载荷
   - `buildYingyongbaoUpdatePayload`
   - 处理 `deploy_type`
   - 若定时发布，处理 `deploy_time`
4. 对每个包先获取上传信息
   - `POST /get_file_upload_info`
5. 使用返回的 `pre_sign_url` 上传文件
   - `PUT pre_sign_url`
6. 汇总序列号和 md5 后提交更新
   - `POST /update_app`

实现特点：

- 应用宝是“先拿上传信息，再 PUT 直传文件，再调用 update_app”
- 32/64 位包分别上传
- `storeReleaseId` 使用 `appId`

### queryReview 链路

入口：

- `queryYingyongbaoReview`

调用顺序：

1. 解析 `packageName`
2. 解析 `appId`
   - 优先 `record.storeReleaseId`
   - 否则回退到配置或元数据
3. `POST /query_app_update_status`
4. 根据 `audit_status` 映射内部状态

状态判定：

- `2` -> `REJECT`
- `3` -> `PASS`
- `8` -> `OFFLINE`
- 其他 -> `AUDITING`
- 驳回原因优先取 `audit_reason`

## 6. 商店差异对比

| 商店 | token 类型 | submit 前置查询 | 文件上传方式 | 提审接口 | 审核查询接口 | `storeReleaseId` |

## 7. 状态流转总结

发布记录的典型状态流转如下：

1. `DOWNLOADING`
2. `DOWNLOAD_SUCCESS`
3. `API_PENDING`
4. `AUDITING`
5. 终态之一
   - `PASS`
   - `REJECT`
   - `OFFLINE`

异常分支：

- 安装包准备失败：`DOWNLOAD_FAIL`
- 商店提审异常：直接转 `REJECT`

审核轮询收口逻辑：

- 只有 `PASS` / `REJECT` / `OFFLINE` 会被视为 finished
- finished 状态会写入 `finishTime`

## 8. 当前结论

当前系统的实现特征如下：

1. 发布主流程统一编排
   - 统一由 `ReleaseOrchestrationService + TokenService + ConfigurableStorePublisher` 驱动
2. 商店差异下沉到独立 publisher
   - 每个商店单独实现 `refreshToken` / `submitRelease` / `queryReview`
3. 统一备包，商店并发提审
   - 当前不是串行逐店提交
4. 灰度能力不是全商店通用
   - 当前仅 `vivo`、`huawei`、`rongyao`、`sanxing` 走 API 灰度链路
5. 请求日志和聚合日志都保留
   - 便于排查第三方接口失败、字段缺失、状态映射异常等问题
6. 文档已按当前代码更新
   - 包含荣耀渠道
   - 包含 OPPO `multi-info -> info` 回退逻辑
   - 包含并发提审而非串行提审的实际行为
